How To Use

RAGTAG Viewer Guide

A practical walkthrough of the local viewer: how to launch it, load an IFC, move between the graph and 3D model pages, inspect elements, run questions, and understand what the app can and cannot tell you.

This guide reflects the current local FastAPI viewer in src/rag_tag/viewer.

Open Graph Open 3D Model Relation Guide

Overview

What This App Does

The viewer is a local workspace for exploring one IFC dataset through three connected perspectives: a WebGL graph view, a raw 3D IFC model view, and a relation guide. It also includes a natural-language query panel backed by the main RAGTAG query pipeline.

Graph View

Default page

Explore nodes and relations, filter edge families, focus elements, and inspect graph structure around the active dataset.

3D Model View

IFC geometry

Open the IFC model itself, search for elements, and inspect the currently selected object without leaving the viewer.

Query Runner

Shared pipeline

Ask natural-language questions. The response shows the chosen route, the answer, and optional structured details from the underlying tools.

Legend Guide

Reference

Read what each relation family means, how viewer modes group edges, and when each relation type is useful.

Start

How to Launch It

You can open the viewer against an existing dataset, or point it at a raw IFC and let the viewer build its runtime artifacts before the server starts.

Launch from an IFC

Builds artifacts 
uv run rag-tag-viewer --ifc ./IFC-Files/Building.ifc

This builds JSONL, SQLite, the debug graph HTML, and the WebGL graph assets before serving the app.

Launch from an existing dataset

Fastest startup 
uv run rag-tag-viewer --graph-dataset Building-Architecture

Use this when your `.jsonl`, `.db`, and viewer assets already exist in `output/`.

Launch with explicit DBs

Multi-DB support 
uv run rag-tag-viewer --db ./output/A.db --db ./output/B.db

The viewer can expose one or more SQLite databases. If a single DB is present, its stem becomes the active dataset automatically.

Useful options

Server + payload
  • --host and --port choose where the local server runs.
  • --graph-payload-mode minimal keeps graph payloads lighter.
  • --graph-payload-mode full exposes fuller graph payloads if you need them.
Open the app in a browser after startup. The default graph page is served at /, the model page at /model, and this guide at /how-to-use.

Workflow

Typical End-to-End Flow

Most sessions follow the same loop: load a model, inspect the graph, search for elements, open the 3D view when needed, then ask targeted questions in the query runner.

  1. Start with an IFC or an existing dataset

    If you launch with --ifc, the viewer builds everything it needs up front. If you launch against existing output files, it loads the detected dataset and graph assets directly.

  2. Upload a new IFC if you want to rebuild in-place

    The Upload IFC button on the graph and model pages sends the file to the server, starts a build job, shows progress, and refreshes the viewer when the new artifacts are ready.

  3. Explore the graph

    Use the graph modes, legend, explorer, and inspector to understand structure, class distribution, and relation slices before asking more complex questions.

  4. Switch to the 3D model when geometry matters

    The 3D view helps when you need the raw IFC model context instead of the abstracted relation graph.

  5. Ask natural-language questions

    The query runner uses the main RAGTAG pipeline. You can keep a local transcript, inspect the chosen route, and copy the conversation.

Graph Page

Using the Graph Viewer

The graph page is the main workspace. It combines a large WebGL graph canvas with filtering controls, side panels, and the query dock.

Graph modes

  • Full Graph shows the broadest available context.
  • Nodes emphasizes node structure.
  • Edges emphasizes connections.
  • Hierarchy, Spatial, Topology, and Explicit IFC focus on relation families.

Display toggles

  • Edge Labels shows or hides relation labels.
  • BBoxes overlays available bounding boxes.
  • Meshes overlays exported mesh data when present.
  • Legend, Explorer, and Inspector show or hide the side panels.

Utility buttons

  • Fullscreen expands the graph stage.
  • Reset view fits the graph back into view.
  • Relation guide opens the explainer page in a new tab.
  • Legacy Plotly debugger opens the older debug graph HTML.

When the graph is empty

If no IFC or graph assets are available, the graph stage shows an empty-state panel explaining what is missing and how to recover.

Side Panels

Search and Inspect Elements

Both the graph and model pages expose a search-driven element browser plus an inspector for the currently selected element.

Node Explorer

  • Search by label, IFC class, GUID, or element identifier.
  • Filter by IFC class and change page size.
  • Move through paginated results with Previous and Next.
  • Use Focus to jump the graph or model to an element.

Inspector

  • Select a node in the graph, select an object in the 3D model, or focus a search result.
  • The inspector shows a JSON-like payload with label, class, identifiers, and properties.
  • Element lookup supports direct node IDs, GlobalId values, and Express IDs.

3D View

Using the IFC Model Viewer

The model page is paired with the graph page and uses the currently active IFC file when one is available.

What it needs

The graph view can still work from generated artifacts alone, but the 3D page needs an active raw IFC file. If none is available, the page shows an empty state explaining how to provide one.

Main controls

  • Switch back to the graph page from the top nav or graph shortcut.
  • Use fullscreen for a larger modeling workspace.
  • Use reset view to fit the model back into frame.
  • Use the same explorer and inspector pattern as the graph page.

Queries

Using the Query Runner

The bottom dock on the graph page lets you ask natural-language questions about the active dataset and review the response history.

How to ask

  • Type a question into the text area and press Run Query.
  • Press Enter to submit quickly, or use line breaks when needed.
  • Each response is appended to the running transcript.

What the response shows

  • The chosen route, such as SQL or graph.
  • A short routing decision summary.
  • The answer text plus optional samples, list rows, warnings, or details.

Details and transcript

  • Details: On shows the structured JSON details block.
  • Copy Transcript copies the full session transcript in a Markdown-friendly format.

How routing works

Deterministic counts and aggregations are intended to come from SQL, while spatial, adjacency, containment, and topological reasoning are handled through the graph path.

Artifacts

What Gets Built Behind the Scenes

Uploading or launching from an IFC does more than load a single file. The viewer depends on several generated artifacts that support different parts of the app.

JSONL export

Parsed IFC content is exported into JSONL so the graph and SQL stages can work from a normalized intermediate format.

SQLite database

SQL-backed counts, lists, and aggregations rely on the generated or discovered SQLite database for the active dataset.

Graph runtime + WebGL bundle

The interactive graph page loads a manifest plus binary assets for nodes, edges, and optional overlays such as bounding boxes and meshes.

Legacy debug graph

The older Plotly HTML graph is still available as a debugging aid and can be opened from the graph toolbar.

Limits

Known Limitations

A few important constraints matter when you interpret what the viewer shows. These are part of the current system design, not user error.

Adjacency is approximate

The current graph uses centroid-distance-based adjacency, not true topological contact detection.

Graph backend is still NetworkX

There is no Neo4j or Cypher backend yet, so graph exploration stays within the current in-process runtime.

3D view depends on raw IFC availability

If you only have generated graph or DB artifacts but no IFC file, the graph page can still work while the model page cannot.

Query quality follows dataset quality

Missing geometry, incomplete IFC relationships, or absent assets can reduce what the graph and query layers are able to answer cleanly.
